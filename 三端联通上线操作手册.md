# 三端联通上线操作手册

本文用于把“日程安排”的云同步服务、Windows 电脑端、手机端 App/PWA 串成一条可验收的上线流程。

## 1. 当前架构

- 云同步服务端：`D:\apps\schedule_sync_server`，服务器部署到 `/opt/apps/schedule-sync`。
- Windows 电脑端：`D:\apps\schedule\schedule_backend` + `D:\apps\schedule\schedule_frontend`，本地 SQLite 离线优先。
- 手机端：`D:\apps\schedule\schedule_mobile`，Vue 3 + Capacitor，IndexedDB 离线优先。
- 公网入口：复用服务器 `/opt/apps/hiremate/Caddyfile` 与 `shared_gateway` 网络。
- 正式同步域名：`schedule-sync.zenithy.art`。
- 同步协议：`sync envelope v1`，核心字段为 `entity_type / sync_id / sync_version / sync_deleted / relation_sync_ids / data`。
- 不同步内容：AI API Key、Token、本地运行配置、服务器私密配置。
- 手机端已完成最终功能性基线：模板自动刷新每日任务、AI 保留单独开始时间、日程事件可编辑删除、push 成功后本地实体标记为已同步。

## 0. 手机端冻结规则

- `schedule_mobile` 已封板为最终功能基线。
- 后续上线云服务器、维护电脑端、更新 Caddy、扩展管理后台或调整同步协议时，默认不得修改手机端目录。
- 手机端只允许 iOS 打包发布、严重故障紧急修复、云端无法兼容旧手机端时的最小协议修复。
- 新业务功能优先在电脑端和云服务器端实现；云端必须保持旧手机端可忽略未知字段、可读取核心字段、不崩溃。
- 如果必须改手机端，改后必须执行 `npm test -- --run`、`npm run build`、`npm run cap:sync`。
- 自动同步策略：本地改动后约 3 秒防抖上传，前台每 45 秒拉取云端更新；每日任务状态采用后改覆盖。

## 2. 云服务器部署

### 2.1 准备代码

在本地把 `D:\apps\schedule_sync_server` 推送到独立 Git 仓库。服务器上固定目录：

```bash
sudo mkdir -p /opt/apps
cd /opt/apps
git clone <你的 schedule_sync_server 仓库地址> schedule-sync
cd /opt/apps/schedule-sync
```

### 2.2 配置环境变量

```bash
cp .env.example .env
nano .env
```

必须修改：

- `APP_ENV=production`
- `APP_BASE_URL=https://schedule-sync.zenithy.art`
- `SCHEDULE_SYNC_DOMAIN=schedule-sync.zenithy.art`
- `POSTGRES_USER=autsky`
- `POSTGRES_PASSWORD=<强密码>`
- `DATABASE_URL=postgresql+psycopg://autsky:<URL编码后的强密码>@postgres:5432/schedule_sync`
- `JWT_SECRET=<至少 32 位随机字符串>`
- `ALLOWED_ORIGINS` 加入正式域名、Tauri 和 Capacitor 来源。
- `PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple`
- `PIP_TRUSTED_HOST=pypi.tuna.tsinghua.edu.cn`
- `APT_MIRROR=https://mirrors.tuna.tsinghua.edu.cn/debian`
- `APT_SECURITY_MIRROR=https://mirrors.tuna.tsinghua.edu.cn/debian-security`

推荐 `ALLOWED_ORIGINS`：

```env
ALLOWED_ORIGINS=http://127.0.0.1:1420,http://localhost:1420,http://127.0.0.1:5173,http://localhost:5173,http://127.0.0.1:5174,http://localhost:5174,http://localhost,tauri://localhost,capacitor://localhost,ionic://localhost,https://schedule-sync.zenithy.art
```

### 2.2.1 PostgreSQL 用户统一规则

生产环境统一使用邮箱作为 PostgreSQL 用户名：

```env
POSTGRES_USER=autsky
```

`DATABASE_URL` 使用同一个用户名 `autsky`，不需要邮箱 `@` 编码，所以连接串必须使用：

```env
DATABASE_URL=postgresql+psycopg://autsky:<URL编码后的强密码>@postgres:5432/schedule_sync
```

不要再使用旧写法：

```env
DATABASE_URL=postgresql+psycopg://schedule_sync:<强密码>@postgres:5432/schedule_sync
```

如果服务器已经启动过 PostgreSQL，仅修改 `.env` 不会自动创建新角色。请执行：

```bash
cd /opt/apps/schedule-sync
bash deploy/fix-postgres-user.sh
docker compose up -d api
curl http://127.0.0.1:18130/api/health
curl -vk https://schedule-sync.zenithy.art/api/health
```

如果本地健康检查仍提示 `role "schedule_sync" does not exist`，说明 API 容器仍在读取旧 `DATABASE_URL`，继续检查 `/opt/apps/schedule-sync/.env`。

### 2.2.2 云端 build 慢的处理

云端 Docker build 慢通常有三个原因：

1. 拉取 `python:3.11-slim` 基础镜像慢。
2. Debian `apt-get update / install` 默认源慢。
3. Python 依赖从默认 PyPI 下载慢。

当前项目已经在 `Dockerfile` 和 `.env.example` 中固定：

```env
PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
PIP_TRUSTED_HOST=pypi.tuna.tsinghua.edu.cn
APT_MIRROR=https://mirrors.tuna.tsinghua.edu.cn/debian
APT_SECURITY_MIRROR=https://mirrors.tuna.tsinghua.edu.cn/debian-security
```

如果仍然卡在第一步 `FROM python:3.11-slim`，说明是 Docker Hub 基础镜像拉取慢，需要在服务器 Docker daemon 配置 registry mirror，或先手动拉取基础镜像：

```bash
docker pull python:3.11-slim
```

如果卡在 `apt-get update`，确认 `.env` 里保留了上面的 `APT_MIRROR` 和 `APT_SECURITY_MIRROR`。如果卡在 `pip install`，确认 `PIP_INDEX_URL` 是清华源。

### 2.3 启动服务

```bash
cd /opt/apps/schedule-sync
bash deploy/bootstrap-server.sh
docker compose ps
curl http://127.0.0.1:18130/api/health
```

健康检查应返回 `db=ok`。

### 2.4 接入 Caddy

编辑现有网关：

```bash
nano /opt/apps/hiremate/Caddyfile
```

加入：

```caddy
schedule-sync.zenithy.art {
    import common_site

    reverse_proxy schedule-sync-api:8000
}
```

这里的 `8000` 是 `schedule-sync-api` 容器内部端口；`18130` 只用于服务器本机预览 `curl http://127.0.0.1:18130/api/health`，不要写进 Caddy 反代目标。

确认 Caddy 和同步服务在同一个 Docker 网络 `shared_gateway`，然后：

```bash
docker exec hiremate-caddy caddy reload --config /etc/caddy/Caddyfile
curl https://schedule-sync.zenithy.art/api/health
```

### 2.5 TLSV1_ALERT_INTERNAL_ERROR 专项排查

如果 Windows 电脑端“注册并登录”时报：

```text
无法连接云同步服务器：[SSL: TLSV1_ALERT_INTERNAL_ERROR] tlsv1 alert internal error
```

当前结论应优先判断为云服务器 Caddy/证书层问题，不是电脑客户端业务代码问题。原因是这个错误发生在 HTTPS 握手阶段，还没有进入 `/api/auth/login` 或 `/api/health`。

先在任意电脑上验证：

```powershell
curl.exe -vkI https://schedule-sync.zenithy.art/api/health
```

如果这里也失败，而 `https://zenithy.art`、`https://hiremate.zenithy.art` 正常，就说明只有 `schedule-sync.zenithy.art` 的 Caddy 证书/站点未就绪。

登录服务器后执行：

```bash
cd /opt/apps/schedule-sync
bash deploy/diagnose-gateway.sh
```

手动修复顺序：

```bash
cd /opt/apps/schedule-sync
docker compose ps
curl http://127.0.0.1:18130/api/health

grep -n "schedule-sync.zenithy.art" /opt/apps/hiremate/Caddyfile
docker network inspect shared_gateway | grep -E "hiremate-caddy|schedule-sync-api"

docker network connect shared_gateway schedule-sync-api || true
docker exec hiremate-caddy caddy validate --config /etc/caddy/Caddyfile
docker exec hiremate-caddy caddy reload --config /etc/caddy/Caddyfile
docker logs hiremate-caddy --tail=200

curl -vkI https://schedule-sync.zenithy.art/api/health
```

判断结果：

- `curl http://127.0.0.1:18130/api/health` 失败：同步服务容器没跑好，先修 `schedule-sync`。
- `18130` 成功但 HTTPS TLS 失败：Caddy 没有拿到 `schedule-sync.zenithy.art` 证书，检查 Caddy reload 和 ACME 日志。
- HTTPS 握手成功但返回 `502`：证书好了，但 Caddy 连不上 `schedule-sync-api:8000`，检查 `shared_gateway` 网络。
- HTTPS 返回 `200` 且 `db=ok`：云端正常，再回电脑客户端登录。

## 3. 云端冒烟测试

在服务器运行：

```bash
cd /opt/apps/schedule-sync
bash deploy/smoke-test.sh http://127.0.0.1:18130
```

需要通过：

- 注册账号 A/B。
- 账号 A 登录两台设备。
- A 设备 push，A 另一设备 pull 可见。
- 账号 B pull 不到账号 A 数据。

## 4. Windows 电脑端联通

### 4.1 启动本地端

```powershell
cd D:\apps\schedule
.\start-dev.ps1
```

如果已有 Rust，会进入 Tauri 模式；否则进入 Web fallback。

### 4.2 配置云同步

打开“设置 / 云同步”：

1. 云服务器地址已默认内置为 `https://schedule-sync.zenithy.art`，一般不用手动填写。
2. 填邮箱、密码、设备名。
3. 点击“注册并登录”或“登录”。
4. 点击“手动同步”。

如果失败，先看设置页“最近错误”。常见错误：

- `HTTP 401`：Token 过期或账号密码错误，重新登录。
- `设备未登记`：重新登录，客户端会重新注册设备。
- `Failed to fetch / CORS`：检查云端 `ALLOWED_ORIGINS` 和 Caddy HTTPS。
- `cloud rejected`：存在版本冲突，队列不会被误删，稍后在冲突区处理。

## 5. 手机端 PWA / App 联通

### 5.1 Windows 阶段浏览器预览

```powershell
cd D:\apps\schedule\schedule_mobile
npm install
npm run dev
```

用 Chrome DevTools 切到手机尺寸访问本地地址。

### 5.2 手机端登录同步

打开手机端“同步设置”：

1. 服务器地址已默认内置为 `https://schedule-sync.zenithy.art`，一般不用手动填写。
2. 使用与电脑端相同账号登录。
3. 登录成功后会自动执行 bootstrap。
4. 页面顶部同步状态应显示“已同步”或“待同步 0 条”。

手机端支持：

- 登录后自动 bootstrap。
- App 启动时尝试同步。
- 网络恢复时自动同步。
- App 回到前台时尝试同步并刷新固定模板生成的每日任务。
- Token 过期时自动 refresh 后重试一次。
- 本地 dirty 数据遇到远端变化时进入冲突记录。
- 云端 push 返回 `accepted_items` 后，手机端会更新本地 `sync_version/last_synced_at` 并清除 dirty 状态。
- AI 解析支持只有开始时间的事件；日程页可编辑、删除事件。

## 6. iOS 打包测试

iOS 真机或模拟器阶段必须使用 Mac + Xcode。

### 6.1 从 Git Clone 到 Xcode 打开

在 Mac 上准备代码：

```bash
cd ~/Code
git clone <你的 schedule 仓库地址> schedule
cd schedule/schedule_mobile
npm install
npm test -- --run
npm run build
```

如果仓库里还没有 `ios/` 工程：

```bash
npx cap add ios
npx cap sync ios
npx cap open ios
```

如果已有 `ios/` 工程：

```bash
npx cap sync ios
npx cap open ios
```

### 6.2 安装到你的 iPhone

在 Xcode 中按顺序操作：

1. 左侧选择 `App` 项目。
2. 进入 `Signing & Capabilities`。
3. `Team` 选择你的 Apple ID 或 Apple Developer Team。
4. `Bundle Identifier` 建议使用唯一值，例如 `art.zenithy.schedule`。
5. 用数据线连接 iPhone，手机上点击“信任此电脑”。
6. iPhone 如果提示开发者模式，进入 `设置 / 隐私与安全性 / 开发者模式` 打开并重启。
7. Xcode 顶部设备选择你的 iPhone。
8. 点击左上角 Run，Xcode 会把 App 安装到手机。
9. 第一次运行如果提示未信任开发者，到 iPhone `设置 / 通用 / VPN 与设备管理` 信任证书。

在 Xcode 中重点检查：

- Bundle Identifier。
- Signing Team。
- 网络权限和 HTTPS 域名。
- 登录、同步、退出登录。

上线前建议把 AI Key 从 Capacitor Preferences 升级到 Keychain 插件；当前版本已保证 AI Key 不进入云同步 payload。

### 6.3 App 后续怎么更新

iOS 普通 App 不能像电脑软件一样自己下载新版 IPA 并静默安装。正确路线是：

1. 开发自用：Mac 上 `git pull` 最新代码，执行 `npm install`、`npm run build`、`npx cap sync ios`，然后在 Xcode 点 Run 重新安装到你的 iPhone。
2. 小范围测试：用 Xcode `Product / Archive` 打包上传 App Store Connect，通过 TestFlight 分发；手机端在 TestFlight App 中更新。
3. 正式上线：提交 App Store 审核，用户通过 App Store 更新。

App 内可以做“检查更新”：

- 请求自建版本接口或静态 JSON。
- 提示用户有新版本。
- 跳转到 TestFlight 或 App Store。

App 内不能直接自动下载并安装新版安装包。企业签名、描述文件分发、第三方热更新服务不建议第一版采用，维护和审核风险都更高。

## 7. 三端数据验收

按以下顺序验收：

1. 电脑端新建今日任务，手动同步，手机端同步后可见。
2. 手机端新建长期任务和子任务，手动同步，电脑端同步后可见。
3. 手机端完成学习任务并生成学习记录，电脑端学习记录页可见科目时长。
4. 电脑端删除任务，同步后手机端不再显示。
5. 同一个账号两端可见；另一个账号完全不可见。
6. 电脑端和手机端离线各自新增数据，恢复网络后都能上传。
7. 同一实体两端离线同时修改，恢复网络后至少一端出现清晰冲突提示，不应静默覆盖。
8. 电脑端新增固定任务模板并同步后，手机端打开 Today 会自动生成当天任务，重复打开不重复生成。
9. 手机端输入“明天八点开会”并应用为事件，电脑端同步后应能看到明天 `08:00` 的事件，结束时间可为空。
10. 手机端点击完成每日任务后，电脑端保持前台，45 秒内应自动变为完成；反向取消完成同理。
11. 两端几乎同时改同一每日任务状态时，最终以 `updated_at` 较新的状态为准，但标题和备注不应被状态补丁覆盖。

## 9. 常见问题

- 云端启动失败：检查 `.env` 是否还包含 `replace-with` 或 `sync.example.com`。
- HTTPS 访问失败：检查域名 DNS、Caddy 配置、容器网络 `shared_gateway`。
- 手机端 CORS：检查 `ALLOWED_ORIGINS` 是否包含 `capacitor://localhost`、调试地址和正式域名。
- 同步后没有数据：确认两端登录的是同一个账号，并检查 `latest_change_id` 是否更新。
- AI 不可用：AI Key 是用户端本机配置，不在云端配置，也不会同步。
