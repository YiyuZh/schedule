# 云同步服务与本地客户端构建方案

## 1. 文档定位

本文件不再承载所有细节，而是作为“三端整合联通总指南”。

细节请分别阅读：

1. `手机端构建方案.md`
2. `云服务器构建方案.md`
3. `本地电脑客户端构建方案.md`
4. `手机端UI设计模型Prompt.md`

本文件回答的问题是：

- 手机端、云服务器、本地电脑端分别完成后，怎么连起来？
- 应该按什么顺序联调？
- 怎么确认账号隔离、安全同步和 AI Key 处理没有问题？
- 上线前还要检查什么？

截至 2026-04-24，云同步服务端源码已独立为：

```text
D:/apps/schedule_sync_server
```

当前服务端已具备生产前置硬化：生产配置校验、数据库健康检查、注册/登录限速、显式设备注册、`push` accepted/rejected 明细、Alembic 初始迁移、Docker/Caddy 部署脚本和业务冒烟脚本。桌面端也已补齐“注册并登录”和云端 rejected 时保留本地同步队列。

## 2. 总体架构

```text
Windows 桌面端
  Vue/Tauri
  本地 FastAPI
  本地 SQLite
  本地 AI Key
  Sync Engine
        │
        ▼
云同步服务
  FastAPI
  PostgreSQL
  users / devices / tokens
  sync_records / sync_changes
        ▲
        │
iPhone App
  Vue Mobile / Capacitor
  本地存储
  本机 AI Key
  Sync Engine
```

核心原则：

- 云服务器只做账号和同步。
- 不同账号的数据通过 `user_id` 隔离。
- 业务逻辑仍在客户端。
- AI Key 只在用户设备本地保存。
- 云服务器不代理 AI 请求。
- 电脑和手机都可以离线使用，联网后同步。

## 3. 三端完成后的启动顺序

### 3.1 先启动云同步服务

先确保：

- `GET /api/health` 正常。
- PostgreSQL 正常。
- HTTPS 正常。
- 注册 / 登录正常。
- 用户 A / B 数据隔离测试通过。

### 3.2 再接入 Windows 桌面端

桌面端配置：

- 云服务器地址。
- 登录账号。
- 当前设备名。
- 同步开关。

先完成桌面端自测：

- 新增任务进入同步队列。
- 手动同步能 push 到云端。
- 清空本地数据库后 bootstrap 能恢复数据。
- AI Key 没有进入云端 payload。

### 3.3 再接入手机 Web/PWA

手机端先不要急着打包 iOS。

先用：

```text
Chrome DevTools 手机尺寸预览
iPhone Safari 局域网访问
```

验证：

- 手机端登录。
- 手机端新增任务。
- 手机端 push 到云端。
- 桌面端 pull 到手机端任务。

### 3.4 最后打包 iOS App

iOS 阶段需要：

- Mac
- Xcode
- Capacitor iOS
- iPhone 真机或 Simulator

验证：

- App 能启动。
- 本地存储可用。
- 安全存储可保存 AI Key。
- 离线新增任务，联网后同步。

## 4. 联调顺序

### 阶段 A：云端独立联调

用 curl / Postman 测：

1. 注册用户 A。
2. 注册用户 B。
3. 用户 A 登录。
4. 用户 B 登录。
5. 用户 A push 一条 `daily_task`。
6. 用户 A pull 能拿到。
7. 用户 B pull 拿不到。

必须先过账号隔离，再接客户端。

### 阶段 B：桌面端联调

1. 桌面端登录用户 A。
2. 新增今日任务。
3. 执行手动同步。
4. 云端 `sync_records` 出现用户 A 的记录。
5. 换一个本地数据库模拟用户 A 第二台设备。
6. bootstrap 能恢复该任务。
7. 登录用户 B，确认恢复不到用户 A 数据。

### 阶段 C：手机端联调

1. 手机端登录用户 A。
2. pull 云端数据。
3. 手机端看到桌面端任务。
4. 手机端完成任务。
5. push 到云端。
6. 桌面端 pull 后看到任务完成。

### 阶段 D：双向离线联调

1. 桌面端断网新增任务。
2. 手机端联网新增任务。
3. 桌面端恢复网络。
4. 两端同步后都看到两条任务。

### 阶段 E：冲突联调

1. 桌面端和手机端同时编辑同一任务标题。
2. 两端同步。
3. 按冲突策略合并或进入冲突列表。
4. 用户可以选择本地 / 云端版本。

## 5. 账号数据隔离验收

必须验证：

- 用户 A 看不到用户 B 的任务。
- 用户 A 看不到用户 B 的学习记录。
- 用户 A 不能通过猜 `entity_id` 获取用户 B 数据。
- 用户 B 的设备不能注册到用户 A 名下。
- bootstrap 只返回当前登录用户数据。
- push 时服务端不使用客户端传入的 `user_id`。

失败一条就不能上线。

## 6. 同步验收

### 6.1 基础同步

- 桌面新增任务，手机能看到。
- 手机完成任务，桌面能看到。
- 桌面新增长期任务，手机能看到。
- 手机新增长期子任务，桌面能看到。
- 学习记录能双向同步。

### 6.2 删除同步

- 删除任务后另一端隐藏或标记删除。
- 删除长期子任务后另一端同步删除标记。
- 删除不是直接物理删除，必须先有 `deleted_at` 或 delete change。

### 6.3 离线同步

- 离线新增任务进入本地队列。
- 恢复网络后自动补传。
- 同步失败有错误提示。
- 不因为同步失败丢本地数据。

## 7. AI Key 验收

必须验证：

- 桌面端 AI Key 只在本地。
- 手机端 AI Key 只在本机安全存储。
- 云端数据库没有 `ai_api_key`。
- 云端日志没有 AI Key。
- 同步 payload 没有 AI Key。
- AI 请求直接从用户端发往 DeepSeek / OpenAI 兼容接口。

可以同步：

- `ai_provider`
- `ai_base_url`
- `ai_model_name`
- `ai_plan_model_name`

不能同步：

- `ai_api_key`
- 加密后的 `ai_api_key`

## 8. 冲突规则

第一版冲突策略：

- 不同字段尽量合并。
- 同字段用 `updated_at` 晚的一方。
- `completed` 优先于旧的 `pending`。
- 学习记录按追加型数据处理。
- 删除与编辑冲突时进入冲突列表。

冲突记录要保存在本地 `sync_conflicts`。

## 9. 上线前检查表

### 9.1 云端

- HTTPS 正常。
- 数据库每日备份正常。
- 防火墙只开放必要端口。
- 日志不包含密码 / token / AI Key。
- 用户隔离测试通过。
- 备份恢复测试通过。

### 9.2 桌面端

- 同步设置页可登录 / 退出。
- 手动同步可用。
- 自动同步可用。
- 离线队列可用。
- AI Key 不上传。
- 旧本地数据能补 `sync_id`。

### 9.3 手机端

- iPhone 尺寸 UI 可用。
- 登录 / 退出可用。
- 本地存储可用。
- AI Key 安全存储可用。
- 离线新增任务可用。
- 恢复网络后同步可用。

## 10. 推荐上线顺序

1. 只给自己使用。
2. 增加第二台设备测试。
3. 邀请 1 - 3 个可信用户。
4. 观察 1 周同步日志。
5. 修复冲突和离线问题。
6. 再扩到 10 - 30 个用户。
7. 稳定后再考虑 TestFlight。
