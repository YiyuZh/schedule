# 日程安排云服务器构建方案

## 1. 目标与定位

本方案用于把“日程安排”的云同步服务部署到你现有的云服务器体系中。

云服务器只负责：

- 账号注册、登录、Token 刷新
- 设备登记与设备状态
- 多端同步数据存储
- 变更日志
- 数据备份
- 账号级数据隔离

云服务器不负责：

- AI 调用
- AI Key 保存
- AI 代理
- 学习计时业务
- 日程规划业务
- 桌面端完整业务逻辑

第一版服务名建议固定为：

```text
schedule_sync_server
```

服务器部署目录固定纳入现有 `/opt/apps` 体系：

```text
/opt/apps/schedule-sync
```

公网网关继续复用现有 HireMate 项目里的 Caddy：

```text
/opt/apps/hiremate/Caddyfile
```

这份 Caddyfile 是当前服务器公网入口的事实来源。`gateway-portal` 里的 Caddy 相关文档或示例只能作为参考，不作为实际生效来源。

## 2. 当前服务器事实口径

根据现有部署资料，当前服务器已经采用以下模式：

- 统一项目目录：`/opt/apps/<project>`
- 主页项目目录：`/opt/apps/portal`
- HireMate 生产目录：`/opt/apps/hiremate`
- AI-Interview 生产目录：`/opt/apps/ai-interview`
- 统一公网 Caddy 容器：`hiremate-caddy`
- 统一公网网关配置：`/opt/apps/hiremate/Caddyfile`
- 统一 Docker 网络：`shared_gateway`
- 新项目原则：项目容器不直接占用公网 `80/443`，由 `hiremate-caddy` 统一反代

因此，日程安排云同步服务也必须遵守：

- 项目目录放在 `/opt/apps/schedule-sync`
- 项目自身不再单独启动 Caddy
- API 容器加入 `shared_gateway`
- PostgreSQL 不暴露公网端口
- 正式域名、DNS、`.env`、Caddyfile 必须保持同一口径

## 6. 数据库设计

所有业务数据必须带 `user_id`，不同账号之间绝不能互相读到数据。

### 6.1 users

```text
id
email unique
password_hash
display_name
is_active
last_login_at
created_at
updated_at
```

### 6.2 devices

```text
id
user_id
device_id
device_name
device_type: desktop / ios / web
last_seen_at
created_at
updated_at
```

唯一约束：

```text
unique(user_id, device_id)
```

### 6.3 refresh_tokens

```text
id
user_id
device_id
token_hash
expires_at
revoked_at
created_at
updated_at
```

服务端只保存 Refresh Token hash，不保存明文 Refresh Token。

### 6.4 sync_records

```text
id
user_id
entity_type
entity_id
payload_json
content_hash
version
deleted_at
updated_by_device_id
created_at
updated_at
```

唯一约束：

```text
unique(user_id, entity_type, entity_id)
```

说明：

- `entity_id` 对应客户端本地 `sync_id`
- `payload_json` 保存业务数据快照
- 云端不理解复杂业务细节，只保存和同步结构化 payload

### 6.5 sync_changes

```text
id
user_id
entity_type
entity_id
operation: upsert / delete
version
changed_by_device_id
changed_at
```

索引：

```text
index(user_id, id)
index(user_id, entity_type, entity_id)
index(user_id, changed_at)
```

### 6.6 user_settings

只允许同步非敏感跨端设置：

```text
id
user_id
setting_key
setting_value
value_type
updated_at
```

允许保存：

- `ai_provider`
- `ai_base_url`
- `ai_model_name`
- `ai_plan_model_name`
- 默认任务时长
- 默认同步间隔

禁止保存：

- `ai_api_key`
- 加密后的 `ai_api_key`
- 本地数据库路径
- 本地日志路径
- 本地窗口状态

## 7. API 设计

### 7.1 Health

```http
GET /api/health
```

返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "status": "ok"
  }
}
```

### 7.2 Auth

```http
POST /api/auth/register
POST /api/auth/login
POST /api/auth/refresh
POST /api/auth/logout
GET  /api/auth/me
```

登录请求：

```json
{
  "email": "me@example.com",
  "password": "password",
  "device_id": "desktop-uuid",
  "device_name": "我的电脑",
  "device_type": "desktop"
}
```

登录成功返回：

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "token_type": "bearer",
  "expires_in": 1800
}
```

### 7.3 Devices

```http
GET    /api/devices
POST   /api/devices/register
PUT    /api/devices/{device_id}
DELETE /api/devices/{device_id}
```

### 7.4 Sync

本地桌面端当前第一期同步客户端已经预留了下面路径：

```http
POST /api/sync/push
POST /api/sync/pull
GET  /api/sync/bootstrap
```

`push` 必须：

- 从 Token 解析 `user_id`
- 校验 `device_id` 属于当前用户
- 只写当前用户的数据
- 写入 `sync_records`
- 写入 `sync_changes`
- 返回 accepted / rejected / conflict 结果
- 返回 `accepted_queue_ids`
- 返回 `rejected_items`

`pull` 必须：

- 只返回当前用户的数据
- 支持 `since_change_id`
- 支持 limit，建议单次最多 500 - 1000 条
- 返回 `changes` 和 `latest_change_id`

`bootstrap` 用于新设备首次全量拉取：

- 只返回当前用户的数据
- 支持分页
- 不返回其他用户任何数据

## 8. 账号隔离规则

所有数据库查询必须遵守：

```sql
where user_id = current_user.id
```

禁止：

- 由客户端上传 `user_id` 决定归属
- 不带 `user_id` 查询同步数据
- 管理员接口默认返回用户明细

验收：

- 用户 A push 的任务，用户 B pull 不到
- 用户 A bootstrap，不会得到用户 B 的任何数据
- 用户 B 猜测用户 A 的 `entity_id` 也无法读取

## 9. Docker Compose 部署方案

`schedule-sync` 项目自己只放 API 和 PostgreSQL，不放 Caddy。

服务器目录：

```text
/opt/apps/schedule-sync/
├─ docker-compose.yml
├─ Dockerfile
├─ .env
├─ logs/
├─ backups/
└─ app/
```

示例 `docker-compose.yml`：

```yaml
services:
  postgres:
    image: postgres:16
    container_name: schedule-sync-postgres
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${POSTGRES_DB:-schedule_sync}
      POSTGRES_USER: ${POSTGRES_USER:-schedule_sync}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?Set POSTGRES_PASSWORD in .env}
    volumes:
      - schedule_sync_postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-schedule_sync} -d ${POSTGRES_DB:-schedule_sync}"]
      interval: 10s
      timeout: 5s
      retries: 12

  api:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        PIP_INDEX_URL: ${PIP_INDEX_URL:-https://pypi.tuna.tsinghua.edu.cn/simple}
        PIP_TRUSTED_HOST: ${PIP_TRUSTED_HOST:-pypi.tuna.tsinghua.edu.cn}
    container_name: schedule-sync-api
    restart: unless-stopped
    env_file:
      - .env
    expose:
      - "8000"
    ports:
      - "127.0.0.1:${SCHEDULE_SYNC_PREVIEW_PORT:-18130}:8000"
    volumes:
      - ./logs:/app/logs
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - default
      - shared_gateway

volumes:
  schedule_sync_postgres_data:
    name: schedule_sync_postgres_data

networks:
  shared_gateway:
    external: true
    name: ${SHARED_CADDY_NETWORK:-shared_gateway}
```

说明：

- `127.0.0.1:18130` 只用于服务器本机预览和健康检查，不对公网开放
- 公网访问走 `hiremate-caddy -> schedule-sync-api:8000`
- PostgreSQL 只在 compose 内网使用，不映射宿主机端口

## 11. 统一 Caddy 网关接入

实际要修改的是：

```text
/opt/apps/hiremate/Caddyfile
```

不是 `schedule-sync` 项目里的 Caddyfile，也不是 `gateway-portal` 里的参考文件。

推荐站点块：

```caddy
sync.<your-domain> {
    import common_site

    reverse_proxy schedule-sync-api:8000
}
```

当前项目提供的示例位于：

```text
D:/apps/schedule_sync_server/deploy/caddy.schedule-sync.example
```

它默认使用 `{$SCHEDULE_SYNC_DOMAIN}`，需要确保 `/opt/apps/hiremate/.env` 或实际 Caddy 环境里存在该变量；如果不想使用变量，就直接替换为真实域名。

如果未来需要临时保护 `/docs`：

```caddy
sync.<your-domain> {
    import common_site

    @docs path /docs* /redoc* /openapi.json
    basicauth @docs {
        your-email@example.com <CADDY_HASHED_PASSWORD>
    }

    reverse_proxy schedule-sync-api:8000
}
```

改完后在服务器执行：

```bash
cd /opt/apps/hiremate
docker exec hiremate-caddy caddy fmt --overwrite /etc/caddy/Caddyfile
docker exec hiremate-caddy caddy validate --config /etc/caddy/Caddyfile
docker exec hiremate-caddy caddy reload --config /etc/caddy/Caddyfile
```

如果公网 502，优先检查：

```bash
docker network inspect shared_gateway | grep -E "hiremate-caddy|schedule-sync-api"
docker logs --tail=100 schedule-sync-api
docker exec hiremate-caddy caddy validate --config /etc/caddy/Caddyfile
```

不要一上来把反代目标改成宿主机 `127.0.0.1:18130`。当前服务器的网关事实是 Docker Caddy 容器，正确模式是通过 `shared_gateway` 解析容器名。

## 12. Git 与 Deploy Key 流程

多项目服务器不要覆盖全局 `Host github.com`。日程同步服务应使用项目级 SSH alias。

服务器生成 deploy key：

```bash
ssh-keygen -t ed25519 -C "schedule-sync deploy key" -f ~/.ssh/id_ed25519_schedule_sync
cat ~/.ssh/id_ed25519_schedule_sync.pub
```

把 `.pub` 输出粘贴到 GitHub 仓库：

```text
Settings -> Deploy keys -> Add deploy key -> Allow read access
```

服务器 SSH config：

```bash
cat >> ~/.ssh/config <<'EOF'
Host github.com-schedule-sync
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_schedule_sync
    IdentitiesOnly yes
EOF
chmod 600 ~/.ssh/config
```

验证：

```bash
ssh -T git@github.com-schedule-sync
```

看到类似下面内容就是成功，不是报错：

```text
You've successfully authenticated, but GitHub does not provide shell access.
```

首次克隆：

```bash
cd /opt/apps
git clone git@github.com-schedule-sync:YiyuZh/schedule-sync-server.git schedule-sync
cd /opt/apps/schedule-sync
cp .env.example .env
```

注意区分：

- 仓库 SSH 地址：`git@github.com:YiyuZh/schedule-sync-server.git`
- deploy key 公钥内容：`cat ~/.ssh/id_ed25519_schedule_sync.pub`

这两个不是同一个东西。

## 15. 备份与回滚

### 15.1 发布前备份

统一备份根目录：

```text
/opt/apps/backups/releases/$RELEASE_ID
```

示例：

```bash
export RELEASE_ID=schedule-sync-$(date +%Y%m%d-%H%M%S)
mkdir -p /opt/apps/backups/releases/$RELEASE_ID

cp /opt/apps/schedule-sync/.env /opt/apps/backups/releases/$RELEASE_ID/schedule-sync.env
cp /opt/apps/hiremate/Caddyfile /opt/apps/backups/releases/$RELEASE_ID/Caddyfile

docker exec schedule-sync-postgres pg_dump -U schedule_sync schedule_sync \
  > /opt/apps/backups/releases/$RELEASE_ID/schedule_sync.sql
```

禁止执行：

```bash
docker compose down -v
```

### 15.2 每日数据库备份

建议目录：

```text
/opt/apps/backups/schedule-sync/postgres
```

命令：

```bash
mkdir -p /opt/apps/backups/schedule-sync/postgres
docker exec schedule-sync-postgres pg_dump -U schedule_sync schedule_sync \
  > /opt/apps/backups/schedule-sync/postgres/schedule_sync_$(date +%F).sql
```

保留策略：

- 最近 7 天每日备份
- 最近 8 周每周备份
- 每月手动下载一份到本地

### 15.3 回滚步骤

```bash
cd /opt/apps/schedule-sync
git log --oneline -5
git checkout <LAST_GOOD_COMMIT>
docker compose up -d --build
curl -fsS http://127.0.0.1:18130/api/health
```

如果 Caddyfile 也改坏了：

```bash
cp /opt/apps/backups/releases/$RELEASE_ID/Caddyfile /opt/apps/hiremate/Caddyfile
cd /opt/apps/hiremate
docker exec hiremate-caddy caddy validate --config /etc/caddy/Caddyfile
docker exec hiremate-caddy caddy reload --config /etc/caddy/Caddyfile
```

如果需要恢复数据库，先停 API，再恢复 dump。恢复前必须再次确认备份文件正确，避免覆盖有效生产数据。

## 16. 业务级冒烟测试

上线前必须通过：

- `GET /api/health` 正常
- 注册用户 A
- 注册用户 B
- 用户 A 登录并注册桌面设备
- 用户 B 登录并注册桌面设备
- 用户 A push 一条 `daily_task`
- 用户 A 另一设备 pull 到该任务
- 用户 B pull 不到用户 A 的任务
- 用户 A 删除任务后，用户 A 另一设备收到 delete change
- 用户端 AI Key 不出现在云端任何 payload

建议 curl 验证：

```bash
curl -fsS https://sync.<your-domain>/api/health
```

完整同步链路建议用桌面端设置页验证：

1. 填写云服务器地址
2. 登录账号
3. 新增今日任务
4. 点击“同步数据”
5. 检查云端 `sync_records`
6. 另一设备 pull

## 17. 2 核 2G 容量预估

前提：

- 云端只做账号和同步
- 不跑 AI
- 不代理 AI
- 不保存大文件
- PostgreSQL 与 API 同机 Docker 部署

保守范围：

```text
注册用户：100 - 300 人
日活用户：30 - 100 人
同时在线：10 - 30 人
瞬时请求：10 - 30 RPS
同步记录：10 万 - 80 万条
```

乐观范围：

```text
注册用户：500 - 1000 人
日活用户：100 - 300 人
同时在线：30 - 80 人
瞬时请求：30 - 80 RPS
同步记录：100 万条左右
```

粗算：

```text
100 日活 * 2 设备 * 每 30 秒 push+pull 2 次 / 30 = 约 13.3 RPS
```

建议配置：

```text
uvicorn workers: 1 - 2
PostgreSQL max_connections: 30 - 50
应用连接池: 5 - 10
同步默认间隔: 30 - 60 秒
单次 pull 限制: 500 - 1000 条
```

需要升级服务器的信号：

- 日活超过 300
- P95 延迟超过 500ms
- 内存长期超过 80%
- PostgreSQL 连接经常耗尽
- `sync_records` 超过 100 万且查询明显变慢

## 18. 安全要求

必须做到：

- HTTPS
- 密码 hash
- Refresh Token hash
- 登录失败限速
- 账号数据隔离
- 防火墙只开放 22 / 80 / 443
- PostgreSQL 不暴露公网
- 定期备份
- CORS 白名单明确

明确不做：

- 不保存 AI Key
- 不代理 AI 请求
- 不把用户本地数据库整库上传
- 不暴露 PostgreSQL 端口
- 不在 `schedule-sync` 项目里再起第二个公网 Caddy

## 19. 常见故障排查

### 19.1 本机预览通，公网 502

优先查：

```bash
docker compose -f /opt/apps/schedule-sync/docker-compose.yml ps
docker network inspect shared_gateway | grep -E "hiremate-caddy|schedule-sync-api"
docker logs --tail=100 schedule-sync-api
```

常见原因：

- `schedule-sync-api` 没加入 `shared_gateway`
- Caddyfile 反代容器名写错
- API 容器启动失败
- Caddy reload 没执行

### 19.2 `git pull` 失败

优先查：

```bash
cd /opt/apps/schedule-sync
git status --short
git remote -v
ssh -T git@github.com-schedule-sync
```

不要在服务器业务目录里手改源码后强行 pull。需要改代码时，优先在本地改完、commit、push，再在服务器 pull。

### 19.3 Docker 构建依赖冲突

不要只升级单个包碰运气。先在临时环境做干跑：

```bash
python -m pip install --dry-run -r requirements.txt
```

验证一组兼容版本后，再锁回 `requirements.txt`。

### 19.4 国内服务器安装很慢

排查顺序：

1. 看 `Dockerfile` 是否给 pip 配了镜像参数
2. 看 `.env` 是否设置 `PIP_INDEX_URL`
3. 看 `requirements.txt` 是否存在依赖冲突
4. 再决定是否换镜像源

不要把“下载慢”和“依赖冲突”混成一个问题。

## 20. 与主页 Portal 的关系

`schedule-sync` 是同步 API 服务，不是用户访问的产品主页。

因此第一版不建议把它作为普通项目卡片放到 Portal 首页，除非后续提供：

- 同步服务状态页
- 管理后台
- 用户文档页

如果未来要在主页新增入口，按 `D:/apps/gateway-portal/主页维护手册.md` 的流程改：

- 本地改 `D:/apps/gateway-portal/portal/site/index.html` 或项目管理数据
- 本地 `git commit && git push`
- 服务器 `/opt/apps/portal` 执行 `git pull && docker compose up -d --build`

公网路由仍然以 `/opt/apps/hiremate/Caddyfile` 为准。
