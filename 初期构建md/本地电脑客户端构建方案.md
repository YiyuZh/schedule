# 本地电脑客户端构建方案

## 1. 目标

本方案描述当前 Windows 桌面端如何接入云同步。

当前桌面端：

```text
Vue/Tauri -> 本地 FastAPI -> 本地 SQLite
```

改造后：

```text
Vue/Tauri
  -> 本地 FastAPI
  -> 本地 SQLite
  -> Sync Engine
  -> 云同步服务
```

核心原则：

- 桌面端继续本地优先。
- 本地业务逻辑不搬到云端。
- 本地写入成功后再进入同步队列。
- 离线时不影响使用。
- AI Key 继续只保存在本地，不同步到云端。

## 0. 当前落地状态

截至 2026-04-24，Windows 本地客户端已完成“云同步底座一期”：

- 已新增 `sync_state / sync_queue / sync_conflicts`
- 已为 `daily_tasks / long_term_tasks / long_term_subtasks` 增加同步元字段
- 已新增启动时安全迁移 `ensure_sync_schema()`，旧 SQLite 会自动补列并回填 `sync_id`
- 已新增本地 `/api/sync/*` 接口
- 已在 `daily_tasks / long_term_tasks / long_term_subtasks` 的新增、更新、删除后写入 `sync_queue`
- 已在设置页新增“云同步”卡片
- 已在设置页新增“注册并登录”入口
- 已在侧边栏展示云同步状态摘要
- 已让顶栏“同步数据”在本地刷新后按条件触发 `/api/sync/run`
- 云端不可达时会记录 `last_error` 并保留待上传队列，不影响本地功能
- 云端返回 rejected/conflict 时，本地同步队列不会被误删

云同步服务端已独立到 `D:\apps\schedule_sync_server`，并已完成生产硬化；正式服务器部署和真实多设备联调仍需后续执行。本地端暂未做复杂冲突合并 UI，pull 到云端变更时会先记录到 `sync_conflicts`。

## 2. 本地数据库改造

### 2.1 需要同步的表

建议第一版同步：

- `task_templates`
- `daily_tasks`
- `events`
- `courses`
- `study_sessions`
- `long_term_tasks`
- `long_term_subtasks`
- 非敏感 `app_settings`

暂不同步：

- `timer_state`
- `ai_api_key`
- AI 日志
- 导入原始文件内容
- 本地窗口状态

### 2.2 业务表新增字段

每张需要同步的表增加：

```text
sync_id
sync_version
sync_dirty
sync_deleted
last_synced_at
updated_by_device_id
```

说明：

- `sync_id`：跨设备唯一 UUID。
- `sync_version`：云端版本号。
- `sync_dirty`：本地有未上传变更。
- `sync_deleted`：本地删除标记。
- `last_synced_at`：最后同步成功时间。
- `updated_by_device_id`：最后修改设备。

### 2.3 sync_queue

```text
id
entity_type
entity_id
operation: upsert / delete
payload_json
base_version
retry_count
last_error
created_at
updated_at
```

每次业务 CRUD 成功后写入队列。

### 2.4 sync_state

```text
id
server_url
user_email
device_id
device_name
access_token
refresh_token
latest_change_id
last_push_at
last_pull_at
last_error
created_at
updated_at
```

注意：

- token 第一版可以放本地 SQLite，但后续建议接系统安全存储。
- `ai_api_key` 不放这里。

### 2.5 sync_conflicts

```text
id
entity_type
entity_id
local_payload_json
remote_payload_json
base_version
remote_version
status: pending / resolved / ignored
resolution
created_at
updated_at
```

第一版可以先自动处理大多数冲突，但需要保留冲突记录。

## 3. 本地后端新增模块

建议新增：

```text
schedule_backend/app/models/sync.py
schedule_backend/app/schemas/sync.py
schedule_backend/app/api/sync.py
schedule_backend/app/services/sync_client.py
schedule_backend/app/services/sync_service.py
schedule_backend/app/services/sync_mapper.py
schedule_backend/app/core/sync_schema.py
```

职责：

- `sync_client.py`：调用云同步服务。
- `sync_service.py`：本地 push / pull / merge 主流程。
- `sync_mapper.py`：本地模型和云端 payload 转换。

## 4. 本地 API 设计

这些接口只给本地前端调用：

```http
GET  /api/sync/status
POST /api/sync/config
POST /api/sync/login
POST /api/sync/logout
POST /api/sync/run
POST /api/sync/push
POST /api/sync/pull
GET  /api/sync/conflicts
POST /api/sync/conflicts/{id}/resolve
```

### 4.1 状态接口

返回：

```json
{
  "enabled": true,
  "server_url": "https://schedule.example.com",
  "user_email": "me@example.com",
  "device_name": "我的电脑",
  "last_push_at": "...",
  "last_pull_at": "...",
  "pending_count": 3,
  "conflict_count": 0,
  "last_error": null
}
```

### 4.2 手动同步

`POST /api/sync/run` 执行：

1. 刷新 token。
2. push 本地队列。
3. pull 云端变更。
4. merge 到本地。
5. 更新状态。

## 5. 本地写入流程

以完成任务为例：

```text
用户点击完成
  -> 本地 daily_tasks 更新为 completed
  -> 写 sync_queue upsert
  -> 前端立即刷新
  -> 后台同步推送到云端
```

失败时：

- 本地状态不回滚。
- `sync_queue.last_error` 写错误。
- 设置页显示同步失败。
- 下次自动重试。

## 6. 云端拉取合并流程

```text
pull changes
  -> 遍历 change
  -> 找本地 sync_id
  -> 如果本地无 dirty，直接覆盖
  -> 如果本地 dirty，进入冲突处理
  -> 更新 latest_change_id
```

第一版冲突规则：

- 不同字段尽量合并。
- 同字段用 `updated_at` 晚的一方。
- `completed` 不被旧的 `pending` 覆盖。
- 学习记录按追加型数据处理。
- 删除标记优先级高，但如果本地有更新则记录冲突。

## 7. 前端设置页改造

设置页新增“云同步”卡片：

- 云服务器地址
- 登录邮箱
- 密码
- 当前设备名
- 登录 / 退出
- 同步开关
- 手动同步按钮
- 最近同步时间
- 待上传数量
- 冲突数量
- 最近错误

顶部“同步数据”按钮改为：

- 当前页刷新 + 如果已登录云同步，则触发 `/api/sync/run`

## 8. AI 配置处理

桌面端继续使用当前 AI 配置：

- `ai_provider`
- `ai_base_url`
- `ai_api_key`
- `ai_model_name`
- `ai_plan_model_name`

同步规则：

- 可以同步非敏感 AI 配置。
- 不同步 `ai_api_key`。
- 不同步加密后的 `ai_api_key`。
- 云端不代理 AI 请求。

## 9. 自动同步策略

建议：

- 启动时同步一次。
- 每 30 - 60 秒同步一次。
- 网络失败时指数退避。
- 用户点击同步按钮时立即同步。
- 退出应用前尝试 push 一次。

避免：

- 每次编辑都立即打云端。
- pull 全量数据。
- 同步过程中阻塞 UI。

## 10. 本地验收

桌面端同步 MVP 必须通过：

- 登录云同步账号。
- 生成 `device_id`。
- 新增任务后写入 `sync_queue`。
- 手动同步后云端有记录。
- 另一份本地数据库 bootstrap 能恢复记录。
- 账号 A 数据不会出现在账号 B。
- 离线新增任务，联网后补传。
- AI Key 不出现在云端 payload。

## 11. 实施顺序

1. 新增本地 sync 表。
2. 给业务表补同步字段。
3. 实现 sync mapper。
4. 实现 sync client。
5. 实现 sync service。
6. 增加本地 `/api/sync/*`。
7. 设置页增加云同步卡片。
8. 所有 CRUD 写同步队列。
9. 增加冲突记录。
10. 和云端联调。
