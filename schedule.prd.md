《日程安排》本地工具应用 PRD V2.0

版本：V2.0
产品名称：日程安排
运行平台：Windows 10
技术路线：Vue 3 + Tauri + FastAPI + SQLite
产品形态：本地桌面应用（支持本地 AI / API 模型接入）

1. 项目概述
1.1 项目背景

用户需要一个运行在 Windows 10 上的本地日程安排工具，用于管理每日任务、固定刷新任务、学习计时、学习记录、课表导入以及 AI 智能录入与规划。

相比普通待办软件，本项目强调以下能力：

每日固定任务自动刷新
学习任务计时与统计
自然语言输入自动加入行程
导入课表生成周期日程
结合个人习惯的 AI 智能安排

本项目采用 本地优先架构：

前端界面运行在 Tauri 桌面壳中
业务逻辑和 AI 调用由本地 FastAPI 服务承载
数据存储在本地 SQLite 数据库中
在无网络情况下，除 AI 相关能力外，其余核心功能仍可正常使用
1.2 产品定位

“日程安排”是一款 本地个人日程与学习管理工具，适合学生、自主学习者和个人规划用户长期使用。

1.3 核心价值
降低每日重复录入成本
提升任务管理与学习记录效率
让自然语言输入直接转化为任务和日程
让 AI 在已有安排基础上给出合理计划，而不是只做提醒
保持数据本地存储，隐私可控
2. 产品目标
2.1 核心目标

本项目需要实现以下核心目标：

目标一：每日任务自动化

支持创建每日固定任务模板，并在每天自动生成当天任务实例。

目标二：任务与日程统一管理

支持任务、行程、课程统一展示与管理，可增删改查、勾选完成、查看状态。

目标三：学习时长可信记录

支持学习任务计时，自动生成学习记录，并在统计页查看总时长和分项时长。

目标四：自然语言快速录入

用户通过输入一句自然语言，即可解析为任务、事件或安排建议。

目标五：AI 智能规划

结合已有日程、固定任务和个人习惯，为新增事项提供时间安排建议。

2.2 非目标

V2.0 不纳入以下内容：

多端同步
团队协作
云端账号体系
第三方日历双向同步
复杂项目管理功能
手机端 App
3. 目标用户
3.1 主要用户群体
大学生
备考学习用户
个人时间管理需求较强的用户
需要同时管理课程、学习、会议、日常安排的人
3.2 用户特征
希望操作简单，不想频繁点表单
希望固定任务每天自动生成
对学习总时长和分类时长敏感

4. 产品原则
4.1 本地优先

核心数据默认存储于本地 SQLite，本地即可使用基础功能。

4.2 高效录入

高频操作尽量减少步骤，自然语言输入作为主入口之一。

4.3 AI 辅助而非强制

AI 负责解析与建议，最终写入由用户确认。

4.4 数据清晰可追溯

学习时长、任务完成、AI 解析结果都要有记录和来源标识。

4.5 架构可扩展

后续支持更多 AI 模型、导入格式、统计方式和同步能力。

7. 核心概念定义

为了开发时避免混淆，系统中的对象统一定义如下：

7.1 固定任务模板

用于定义每日自动生成的任务规则，例如：

英语学习
健身
背单词

模板本身不代表当天真实任务。

7.2 每日任务实例

由固定模板在某一天生成的真实任务，用户在这一天勾选完成、编辑标题、修改时间等，操作的是实例。

7.3 单次事件

具有明确时间点或时间段的安排，例如：

明天 10 点开会
下午去医院
晚上去网吧
7.4 周期课程事件

由课表导入产生，具备周期规则，例如每周一 8:00-9:35 的英语课。

7.5 学习任务

属于任务或事件中的一种，可绑定学习计时。

7.6 学习记录

一次真实发生的学习会话，例如：

英语学习：19:00 - 20:05，共 65 分钟
7.7 个人习惯

系统基于历史数据统计出的偏好，如：

常在晚上学习英语
周末更倾向下午健身
单次学习时长通常为 45-60 分钟

11. FastAPI 服务设计

这是本次方案改动的重点。

11.1 FastAPI 在项目中的职责

FastAPI 不是可有可无，而是本项目核心业务中台，负责：

提供本地 REST API
统一处理数据库读写
提供任务与日程业务逻辑
提供每日刷新逻辑
提供计时器逻辑
提供 AI 解析与 AI 规划逻辑
提供导入解析能力
提供统计聚合能力
11.2 为什么必须用 FastAPI

相较于把业务全写在前端中，使用 FastAPI 的好处：

第一，业务逻辑集中

固定刷新、计时、统计、冲突检测、导入解析更适合后端统一处理。

第二，AI 接入更清晰

本地模型、OpenAI API、兼容接口都可由 FastAPI 封装，前端无需感知差异。

第三，后续扩展更容易

以后要加同步、通知、备份、日志、模型切换，都可以继续放在服务层。

第四，SQLite 操作更安全

由后端统一管理事务和数据约束，避免前端散乱操作数据库。

13.1 导入目标

17. 验收标准
17.1 固定任务
用户创建“英语学习”模板后，第二天进入应用时自动生成今日任务
同一天不重复生成
17.2 任务管理
支持新增、编辑、删除、勾选完成、标记跳过
操作后前端即时更新
17.3 学习计时
支持开始、暂停、恢复、结束
结束后自动生成学习记录
学习记录页能正确展示该次时长
17.4 学习统计
能正确统计单科累计学习时长
能正确统计总学习时长
17.5 自然语言录入
输入“明天上午10点去开会”后，系统能解析并展示预览
用户确认后写入日程
17.6 课表导入
导入合法 JSON 后，生成对应周期课程
导入前有预览，导入后日程页可查看
17.7 AI 智能安排
输入“今天想去网吧”时，系统能结合已有安排给出建议方案
用户确认后可写入
17.8 FastAPI 服务
前端所有写操作均通过 FastAPI 完成
FastAPI 可正常读写 SQLite 并返回标准响应
18. 风险与应对
风险一：AI 解析不稳定

应对：固定 JSON 输出 + schema 校验 + 用户确认

风险二：计时状态丢失

应对：当前计时状态落库保存，应用重启后恢复

风险三：固定任务重复生成

应对：增加每日生成标记，按日期去重

风险四：导入课表格式混乱

应对：提供标准模板 + 导入预览 + 明确错误提示

风险五：前后端联调混乱

应对：前期先固定 API 协议与字段命名，再分模块开发
